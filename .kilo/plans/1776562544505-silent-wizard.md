## Architecture Rules (Critical — Do Not Break)

### 1. Parsing Chain
```js
const parsed = window.NodeSystem.parse(luaText);       // structured nodes/edges
const graph  = window.NodeSystem.analyze(parsed);      // pipeline, animation, issues
const lua    = window.LuaAnalyzer.analyze(luaText, parsed); // pass parsed to avoid double-parse
```

### 2. Path-Type Filtering (Main Bug Fix)
Fusion creates internal motion-path helper nodes that must **never** appear as canvas nodes. The fix is applied in **three places**:

**Full path type set (10 types — DO NOT REDUCE):**
- `BezierSpline`, `PolyPath`, `Path`, `XYPath`, `BezierPath`, `LinearPath`, `MoPath`, `MotionPath`, `SplinePath`, `PathFollow`

**Locations:**
1. `nodes-system.js` — `PATH_TYPES` Set + `PATH_NAME_RE` regex in second-pass loop of `parseFusion()`
2. `nodegraph.html` — `SKIP_PATH_TYPES` + `SKIP_PATH_NAME_RE` in `parseSettingText()`, and `_IMPORT_PATH_TYPES` + `_IMPORT_PATH_NAME_RE` in `importFromFusion()`
3. `lua-analyzer.js` — `PATH_TYPES` Set used to classify `pathHelpers` separately from real tools

### 4. Import Routing in nodegraph.html
Three parsers called in priority order by `importFromFusionAuto()`:

1. `importFromNodeSystem()` — **preferred path**. Uses `NodeSystem.parse()` → `NodeSystem.normalize()`. Path filtering handled by nodes-system.js itself.
2. `importFromFusion()` — **fallback**. Has own brace-counting parser. Path filtering done locally with `_IMPORT_PATH_TYPES`.
3. `parseSettingText()` — used only for per-node params paste in right panel (not bulk canvas import).

After import, both `importFromNodeSystem` and `importFromFusion` call `LuaAnalyzer.analyze()` and display compact summary in `#gpaste-result`.

### 5. Analysis Panel in effect-modal.html
`populateModal(effect)` receives effect data from Supabase. Field `effect.node_code` holds raw Fusion Lua. If present:

```js
const parsed = NodeSystem.parse(node_code);
const graph  = NodeSystem.analyze(parsed);     // pipeline, stats, animation, issues
const lua    = LuaAnalyzer.analyze(node_code, parsed); // expressions, macros, script blocks
renderAnalysis(container, graph, parsed, lua);
```

`renderAnalysis()` renders:
- Prose summary
- Stat cards (nodes/connections/animated)
- Category chips
- Topological pipeline row
- Animated param list
- Issue notices
- "Lua internals" section (path helpers filtered, custom tools, expressions, script blocks) — hidden until analysis succeeds

#### Step 4: Integrate Timeline Component
**Function:** `emShowNodeParams()` (around line ~1299)

After existing param panel setup, add:
1. Calculate frame range using `SplineEvaluator.getFrameRange(node)`
2. Collect all unique keyframe frames from node params
3. Initialize `TimelineScrubber` in `#em-timeline-container`
4. Set onFrameChange callback to update value display

**Code pattern:**
```javascript
// Get frame range
var range = window.SplineEvaluator.getFrameRange(node);
_emFrameRange = range;

// Collect keyframes
var allKeyframes = [];
// ... traverse node.params to collect keyframe frames ...

// Create timeline
_emTimeline = new window.TimelineScrubber(
  document.getElementById('em-timeline-container'),
  {
    startFrame: range.start,
    endFrame: range.end,
    currentFrame: range.start,
    keyframes: allKeyframes,
    onFrameChange: function(frame) {
      _emCurrentFrame = frame;
      if (_emValueDisplay) {
        _emValueDisplay.setFrame(frame);
      }
      // Update selected param display if needed
    }
  }
);
```
